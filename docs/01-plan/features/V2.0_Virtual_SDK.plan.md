# [Plan] V2.0_Virtual_SDK - Native SDK Hosting (RPC Hub)

## 1. 개요 (Objective)
파이썬 재구현 시 발생하는 '데이터 비대칭' 및 '상태 관리'의 복잡성을 해결하기 위해, 실제 APK 내부의 네이버 지도 SDK를 안드로이드 환경(Worker App)에서 직접 호스팅합니다. 파이썬은 고수준 시나리오 제어만 담당하고, 실제 패킷 생성은 진짜 SDK 엔진이 자율적으로 수행하게 하여 100% 정합성을 보장합니다.

## 2. 시스템 구조 (Architecture)

### 2.1. Android Worker App (Java/Kotlin)
- **Host Engine**: `navermap-sdk.jar`를 로드하여 `NaviSDK`, `NLog` 인스턴스 유지.
- **RPC Server**: 외부(Python) 명령을 수신하는 경량 HTTP/Socket 서버 탑재.
- **Listener Bridge**: 수신된 파라미터(좌표, 이벤트)를 SDK 내부 리스너에 직접 주입.
- **Packet Harvester**: OkHttp 인터셉터를 통해 SDK가 생성한 원본 패킷(Header/Body) 가로채기.

### 2.2. Python Controller (Manager)
- **Scenario Manager**: 주행 경로 및 기기 프로필 관리.
- **Signature Engine**: Harvester로부터 받은 패킷에 `NaverCrypto` 서명을 입혀 최종 전송.
- **Sequence Monitor**: 전송 성공 여부에 따른 시퀀스 정합성 관리.

## 3. 구현 단계 (Roadmap)

### Phase 1: SDK Container 구축
- 안드로이드 환경에서 네이버 SDK v6.4를 에러 없이 초기화하고 메모리에 유지하는 최소형 앱(Service) 개발.

### Phase 2: 이벤트 주입 인터페이스 완성
- `{event: "MOVE", params: {lat, lng}}` 등의 명령을 SDK의 `onLocationChanged`로 연결하는 브릿지 로직 구현.

### Phase 3: 실시간 패킷 추출 파이프라인
- SDK가 헬스체크 및 주행 로그를 쏘는 찰나에 바이너리 데이터를 낚아채어 파이썬으로 전달하는 인터페이스 안정화.

## 4. 기대 효과
- **논리적 무결성**: 60초 주기, 벌크 전송 등 SDK 고유의 판단 로직을 100% 그대로 활용.
- **분석 비용 절감**: 복잡한 Smali 리버싱 없이 SDK의 입/출력단만 제어하여 개발 속도 극대화.
