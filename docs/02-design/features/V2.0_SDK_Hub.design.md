# [Design] V2.0_SDK_Hub - Java Object-Network Initialization Strategy

## 1. 개요 (Objective)
Headless Java 환경(PC JVM)에서 네이버 지도 SDK(`navermap-sdk.jar`)가 정상적으로 동작하기 위해서는, 안드로이드 생태계에서 시스템이 자동으로 해주던 **"객체 간의 연결(Object Dependency Injection)"**을 수동으로 구축해야 합니다. 이 문서는 SDK 내부의 핵심 싱글톤 객체들을 메모리에 올리고 서로 연결하는 **Object-Network (객체 네트워크) 초기화 전략**을 설계합니다.

## 2. 핵심 객체 네트워크 토폴로지 (Core Object Network Topology)

SDK가 완벽한 주행 상태를 유지하기 위해 필요한 4대 핵심 컴포넌트입니다.

### 2.1. Foundation Layer (기반 환경)
- **`MockContext`**: 파일 시스템(`/data/data/...`), `SharedPreferences`, `Resources`를 가상으로 제공하는 허브의 대지(Ground).
- **`NidAppContext`**: NLog와 네비게이션 엔진이 공통으로 의존하는 글로벌 컨텍스트 및 로그인 상태 관리.

### 2.2. Telemetry Layer (통계 및 헬스체크)
- **`NTrackerCore`**: 모든 로깅(Heartbeat, Action Log)의 총괄 매니저.
- **`NTrackerAppLifecycleMonitor`**: 앱의 Foreground/Background 상태를 NTrackerCore에 전달하는 브릿지.
- **`NLog`**: `nlog_id`와 `seq`를 실제로 생성하고 OkHttp를 통해 패킷을 쏘는 실무 객체.

### 2.3. Navigation Layer (주행 물리 엔진)
- **`NaviSDK`**: 네비게이션 모듈의 진입점.
- **`GuideEngine` (or `NaviEngine`)**: 좌표를 받아 도로 상황(POI)을 판단하고 이벤트를 발생시키는 핵심 지능.
- **`LocationManager` (Mocked)**: 가짜 좌표를 `GuideEngine`으로 1초마다 펌핑하는 주입기.

## 3. 초기화 시퀀스 설계 (Bootstrapping Sequence)

Java 리플렉션(Reflection)을 사용하여 의존성 순서대로 객체를 강제 인스턴스화하고 조립합니다.

### Phase 1: Context & Core Booting
1. `MockContext context = new MockContext();` 생성.
2. `NidAppContext.init(context);` 강제 호출.
3. `NTrackerAppLifecycleMonitor(context)` 생성 (이 과정에서 `NTrackerCore` 내부적으로 초기화됨).

### Phase 2: Navigation Engine Booting
1. `NaviSDK.init(context, "[FAKE_CLIENT_ID]")` 호출 시도.
2. 리플렉션으로 `NaviLocationManager` 또는 `GuideEngine` 싱글톤 인스턴스 획득.

### Phase 3: Listener & Interceptor Wiring
1. **Network Wire**: `SDKHunter`를 통해 `OkHttpClient` 필드를 찾아 `PacketCaptureInterceptor`를 `interceptors` 리스트에 Add.
2. **Event Wire (Python Bridge)**: 파이썬에서 `MOVE_TO(lat, lng)` 명령이 오면, Phase 2에서 획득한 `LocationManager`의 `onLocationChanged` 리스너를 직접 Invoke 하도록 브릿지 객체 생성.

## 4. 데이터 대칭성(Data Symmetry) 확보 방안
- **No Translation**: Java Hub는 수신된 좌표를 가공 없이 그대로 SDK 리스너에 던집니다.
- **Raw Export**: SDK가 뱉어낸 `OkHttp.Request`의 Body(Protobuf) 바이트 배열을 디코딩하지 않고 그대로 Base64 인코딩하여 파이썬으로 넘깁니다. (파이썬은 서명만 수행)

## 5. 예상되는 기술적 난제 및 우회책
- **난독화된 내부 클래스 찾기**: `GuideEngine`이나 `LocationListener`의 정확한 Smali 클래스명이 난독화(예: `a.b.c`)되어 있을 수 있습니다.
- **우회책**: 클래스명 대신 구현하고 있는 **인터페이스(Interface)**나 **파라미터 타입(`android.location.Location`)**을 리플렉션으로 전수 조사하여 런타임에 동적으로 바인딩하는 전략(Dynamic Binding)을 사용합니다.
